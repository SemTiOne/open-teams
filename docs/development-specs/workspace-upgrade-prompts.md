# 工作空间升级提示词

## 1. 使用方式

本文件提供可复制给 AI 的升级提示词。提示词用于触发检查、规划、实施、验证和收口，但不替代工作空间强约束：任何升级都必须先确认方案，确认后写入 `task-plans/`，再按节点实施和验收。

## 2. 升级检查

### 检查当前工作空间是否需要升级

```text
请检查当前工作空间是否需要升级到 open-teams 最新模板能力。

要求：
1. 先读取 workspace-assets-index.md、task-completion-checklist.md 和 workspace-config/workspace-version.yaml。
2. 识别当前工作空间类型、已应用能力、本地定制和潜在风险。
3. 只输出升级差异、可选升级项、推荐方案和风险，不要修改文件。
4. 等我确认方案后再进入实施。
```

### 只检查某类能力

```text
请只检查当前工作空间的 workflow skills 是否需要升级。

要求：
1. 对照 skills/README.md、docs/development-specs/workflow-skills-design.md 和 skills/_workflow/。
2. 列出缺失、过时或与 AGENTS.md 不自洽的地方。
3. 给出最小升级方案和验证方式。
4. 不要修改文件，先等我确认。
```

## 3. 升级规划

### 生成可验收升级计划

```text
请基于刚才确认的升级方案，生成 task-plans/ 下的实施计划。

要求：
1. 按节点拆分升级内容，每个节点都要有目标、实施步骤、验证方式和验收标准。
2. 明确哪些业务线本地定制不能覆盖。
3. 计划文档落地后先提交给我确认；我未确认实施计划前不要开始改文件。
4. 每个节点完成后必须等待我验收通过再继续。
```

### 处理模板能力与本地定制冲突

```text
请分析本次升级方案与当前工作空间本地定制的冲突。

要求：
1. 区分模板能力、业务线事实和用户已有改动。
2. 对每个冲突给出保留本地定制、采用模板能力或折中合并三种选项。
3. 给出推荐方案和风险。
4. 不要直接修改文件。
```

## 4. 升级实施

### 按确认方案执行升级

```text
确认按已落地且已由我确认的升级计划执行当前节点。

要求：
1. 只执行当前节点，不跨节点推进。
2. 修改前确认目标文件和当前工作区状态。
3. 完成后运行计划中定义的验证。
4. 将完成说明、验证结果和遗留事项写回 task-plans/。
5. 停下等待我验收。
```

### 继续下一个升级节点

```text
当前节点验收通过，继续下一个升级节点。

要求：
1. 先把上一个节点状态更新为已验收。
2. 将下一个节点状态更新为进行中。
3. 严格按计划执行，不临时扩展范围。
4. 完成后提交节点验收说明。
```

## 5. 验证与收口

### 升级完成前验证

```text
请对本次工作空间升级做完成前验证。

要求：
1. 检查 task-plans/ 中所有节点状态是否已同步。
2. 运行 scripts/validate_template_layout.py。
3. 检查变更范围是否只包含本次升级相关文件。
4. 说明未验证项、风险和是否达到任务完成条件。
5. 不要提交代码，先请求我确认任务完成。
```

### 复盘与变更历史

```text
任务完成，进入复盘和变更历史收口。

要求：
1. 输出本次升级复盘，覆盖背景、方案、关键文件、验证结果和后续注意事项。
2. 等我确认复盘后，补充 change-history/<project>/ 下的任务级记录。
3. 记录已同步资料、未同步资料及原因。
4. 最后给出当前变更范围和验证结果。
```

## 6. 版本元信息维护

### 更新版本能力清单

```text
请根据本次已验收的升级内容，更新 workspace-config/workspace-version.yaml。

要求：
1. 只追加或调整与本次升级真实相关的 applied_capabilities。
2. 更新 last_upgrade_at。
3. 保留业务线本地定制说明，不要用模板示例覆盖。
4. 更新后运行模板校验并说明结果。
```
