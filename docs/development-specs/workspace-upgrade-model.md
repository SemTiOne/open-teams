# 工作空间版本维护与升级模型

## 1. 目标

本模型用于让基于本模板创建的业务工作空间能够通过 AI 交互提示词持续按需升级。升级不是静默覆盖文件，而是由用户用提示词触发，AI 根据当前工作空间版本、已应用能力、本地定制和目标升级内容，先形成方案，再按节点实施、验证和沉淀历史。

## 2. 适用对象

| 对象 | 升级重点 |
| --- | --- |
| 模板仓库 | 维护模板自身的能力基线、workflow skills、文档入口和校验规则 |
| 新建业务工作空间 | 从模板能力中选择适合业务线的基础能力，并替换示例配置 |
| 存量传统工程工作空间 | 保留原有仓库和研发流程，渐进式补入源码映射、文档资产、项目场景 skill 和 workflow 门禁 |

## 3. 版本元信息

`workspace-config/workspace-version.yaml` 是 AI 判断升级基线的入口，记录以下信息：

- `template`：模板身份、版本、修订和来源。
- `workspace`：当前工作空间采用方式、源码接入策略、最近升级时间和本地定制说明。
- `applied_capabilities`：当前已应用的模板能力清单。
- `upgrade_policy`：升级时必须遵守的确认、计划、验收和历史记录策略。
- `upgrade_notes`：升级过程中的额外注意事项。

该文件不是包管理器锁文件，不要求精确表达每个文件的哈希；它用于帮助 AI 快速判断“当前大概具备哪些能力、哪些升级可能适用、哪些本地定制需要保护”。

## 4. 升级触发方式

升级由用户通过自然语言或提示词触发，常见表达包括：

- “检查当前工作空间是否需要升级到最新模板能力。”
- “只升级 workflow skills，不改业务项目文档。”
- “对比当前工作空间和 open-teams 模板，列出可选升级项。”
- “按已确认方案执行升级，每个节点完成后等待验收。”

AI 收到升级请求后，必须遵循当前工作空间的方案确认强约束，不能直接修改文件。

## 5. 升级流程

1. 读取 `workspace-assets-index.md`、`task-completion-checklist.md` 和 `workspace-config/workspace-version.yaml`。
2. 识别当前工作空间类型、已应用能力和本地定制说明。
3. 按需读取目标升级相关的 README、规范、workflow skill、项目场景 skill 或校验脚本。
4. 输出升级差异、推荐方案、风险与节点拆分，等待用户确认。
5. 用户确认后，将方案和实施计划写入 `task-plans/`。
6. 按节点实施升级，每个节点完成后验证并等待用户验收。
7. 升级完成后执行完成判定、复盘确认、资料同步和 `change-history/` 记录。

## 6. 本地定制保护

升级时必须优先保护业务线本地事实和已确认流程：

- 不覆盖 `workspace-config/code-sources.yaml` 中的真实仓库映射。
- 不删除业务线已有 `docs/`、`skills/`、`references/` 资产。
- 不默认改写 `AGENTS.md` 中业务线专属约束。
- 不把模板示例值写入已初始化工作空间。
- 若模板能力与业务线定制冲突，先输出冲突说明和备选方案，由用户决定。

## 7. 能力粒度

升级应按能力粒度执行，而不是按文件盲目覆盖。建议能力粒度包括：

- `workflow-skills`：工作空间级 workflow skill 分层、路由和必备章节。
- `workflow-validation`：workflow skill、入口登记和模板卫生校验。
- `gradual-ai-collaboration-adoption`：存量工程渐进式 AI 协作转型定位和采用路径。
- `workspace-upgrade-mechanism`：版本元信息、升级 workflow、升级提示词和升级校验。

后续新增能力时，应在 `workspace-version.yaml` 中追加 `applied_capabilities`，并在相关入口文档中说明。

## 8. 验收标准

- AI 能通过版本元信息识别当前模板基线和已应用能力。
- 升级过程不会绕过方案确认、计划落档和节点验收。
- 升级方案能明确区分模板能力、业务线本地定制和不在本次范围的内容。
- 升级完成后能运行模板校验，并补充复盘与变更历史。
