# 工作空间版本规则

## 1. 目的

本规范用于说明工作空间模板如何记录用户可读版本历史，以及 `CHANGELOG.md`、`workspace-config/workspace-version.yaml` 和 `change-history/` 的职责边界。

## 2. 版本记录资产

| 资产 | 面向对象 | 职责 |
| --- | --- | --- |
| `CHANGELOG.md` | 工作空间使用者 | 说明每个模板版本新增了什么、升级影响、兼容性和推荐升级提示词 |
| `workspace-config/workspace-version.yaml` | AI 与维护者 | 记录当前工作空间模板版本、已应用能力、升级策略和本地定制说明 |
| `change-history/<project>/` | 维护者与追溯场景 | 记录已完成任务的过程、验证、用户确认和复盘结论 |
| Git commit / PR | 版本维护者 | 记录代码和文档改动的提交历史 |

## 3. 版本号规则

模板版本采用 `MAJOR.MINOR.PATCH`：

- `MAJOR`：工作空间目录结构、约束机制或升级方式发生不兼容变化。
- `MINOR`：新增可选能力、workflow skill、升级机制或重要模板资产，且不强制破坏既有业务工作空间。
- `PATCH`：修正文档、校验脚本、小范围模板缺陷或说明错误。

业务工作空间可以不逐版本追随模板；升级应按能力按需选择，而不是盲目覆盖。

## 4. 版本记录格式

`CHANGELOG.md` 每个版本建议包含：

1. 版本号与日期。
2. 新增能力。
3. 升级影响。
4. 兼容性。
5. 推荐升级提示词。

若版本只包含低风险文案修正，可省略推荐升级提示词，但必须说明影响范围。

## 5. 版本与能力关系

`workspace-version.yaml` 中的 `template.version` 表示当前模板能力基线版本，`applied_capabilities` 表示当前工作空间实际已应用的能力。二者不一定完全等同：

- 模板仓库通常应保持版本和能力清单同步。
- 业务工作空间可以处于较旧模板版本，但单独应用某个新能力。
- 存量工程转型工作空间可以只应用文档、skill 或校验能力的一部分。

因此升级判断应同时读取 `CHANGELOG.md` 和 `workspace-version.yaml`，并结合本地定制说明作出方案。

## 6. 何时更新版本记录

出现以下情况时应更新 `CHANGELOG.md`：

- 新增或调整 workflow skill。
- 新增工作空间升级机制、版本机制或重要治理规则。
- 改变模板初始化流程、目录结构或校验规则。
- 修复会影响使用者判断的模板缺陷。

出现以下情况时应更新 `workspace-version.yaml`：

- 当前工作空间应用了新的模板能力。
- 模板版本或修订标识变化。
- 升级策略、本地定制说明或源码接入策略变化。

## 7. 升级提示词关联

每个重要版本都应在 `CHANGELOG.md` 中提供推荐升级提示词，使业务工作空间用户可以直接触发 AI 检查是否需要升级。提示词必须遵循：

- 先检查和输出方案。
- 不确认不修改。
- 保护本地定制。
- 按节点执行和验收。
- 升级后验证并记录变更历史。
