# AI 工作约束

## 首读要求

在本工作空间内开展任何 AI 工作前，必须优先阅读以下文件：

- [workspace-assets-index.md](./workspace-assets-index.md)
- [task-completion-checklist.md](./task-completion-checklist.md)

其中：

- `workspace-assets-index.md` 是 `sources/` 之外资产的统一入口，包含文档、skills、参考资料、脚本和优先级索引。
- `task-completion-checklist.md` 用于约束每次任务收尾时的完成判定、用户确认、复盘确认和资料同步闭环。

## 执行要求

1. 在未阅读上述索引前，不要直接开始跨目录搜索、写方案或改代码。
2. 若任务已明确落到单项目，先按索引中的优先级定位对应 `docs/`、`skills/`、`references/`，再进入 `sources/<project>/`。
3. 若发现索引与仓库现状不一致，应先更新索引，再继续后续工作。
4. 需要按需选择合适的 skill 进行具体工作指导，先根据当前需求在索引中定位对应项目、对应场景，再读取目标 `skills/<project>/<scene>/` 下的必要内容。
5. `docs/`、`skills/`、`references/` 的读取都必须遵循“按需获取”原则，只读取与当前任务直接相关的最小范围。
6. 禁止一次性将全部文档、全部 skill、全部参考资料整体读入上下文。
7. 若 `SKILL.md` 已能给出当前任务所需约束与流程，则优先继续读取该 skill 明确引用的 `references/` 或 `examples.md`。
8. `change-history/` 属于历史归档层，默认非必读；只有在追溯历史改动、复用既有经验、确认某类问题是否已处理过时，才按需读取。
9. `task-plans/` 属于方案归档层，默认非必读；非用户主动提及，不得自动读取、引用或基于其中内容继续推断当前任务。

## 方案确认强约束

1. 任何任务都必须先与用户确认方案。
2. 在用户未对方案作出明确同意前，不允许进行任何方案落地开发动作，包括但不限于修改代码、新增脚本、调整配置、修改文档以配合开发落地、执行会改变工作区状态的实现性操作。
3. 在方案阶段允许进行的动作仅限于理解需求、按需读取必要资料、分析现状、输出方案、与用户确认方案。
4. 用户确认方案后，方可进入开发落地阶段。
5. 若用户只要求讨论、评估、比较方案或暂未明确批准，不得擅自开始实现。

## 实施计划与节点验收机制

1. 用户明确确认方案后，不能直接进入无计划实施，必须先补充与当前方案配套的实施计划。
2. 实施计划必须按阶段或节点拆解，且每一步都应足够具体。
3. 实施计划一旦确定，后续推进应严格对齐该计划执行；若执行中需要偏离计划，必须先与用户重新确认。
4. 每完成一个实施节点后，必须先向用户说明该节点的完成情况、验证结果、遗留问题与当前验收结论。
5. 每个实施节点都必须等待用户明确同意验收通过后，才可进入下一个实施阶段或下一个节点。

## 代码版本维护机制

1. 每次进入具体开发任务前，先确认目标仓库、目标目录、当前分支和本地工作区状态。
2. 涉及源码仓库的具体开发任务时，默认应基于当前主干切出独立开发分支后再开展实现；若必须留在主分支，需先获得用户明确同意。
3. 涉及代码修改时，必须明确区分用户已有改动、当前任务新增改动、与当前任务无关的历史脏改动。
4. 不得擅自覆盖、回退或清理用户已有改动；如存在冲突风险，必须先与用户确认。
5. 完成任务后，必须检查受影响仓库的变更范围，确认改动是否与任务目标一致。

## 任务完成判定机制

1. 每次完成 `bugfix`、`new-feature-dev`、`security-fix`、`performance-tuning`、`api-refactor`、`feature-refactor` 等任务后，必须先基于当前需求、代码改动、验证结果和未完成项，主动判断任务是否达到“可交付完成”状态。
2. 在 AI 自身判定“已完成”后，不能直接结束流程，必须向用户发起明确确认，说明本次任务完成结论、已完成内容、验证结果、未完成项或风险项。
3. 未经过用户明确确认，不得将任务视为最终完成。

## 复盘与确认机制

1. 用户明确确认任务完成后，必须进入复盘阶段，不能直接结束。
2. 复盘内容至少应包含问题或需求背景、解决方案、为什么这样改、关键代码路径/接口/配置/运行链、验证方式与结果、后续应避免的误区和风险点。
3. 复盘输出后，必须再次与用户进行明确确认。
4. 未经过用户对复盘内容的明确确认，不得进入资料同步阶段。

## 工作空间自成长机制

1. 复盘内容经用户确认后，必须按需同步更新到当前仓库相关资料中，确保文档、skill、参考资料与代码现状保持同步。
2. 资料更新必须遵循最小必要原则，只更新与当前任务直接相关的内容。
3. 需要优先判断应更新的资产类型：
   - 架构变更、模块关系变更：更新 `docs/`
   - 开发约束、实现边界、常见坑：更新 `docs/` 开发规范或 `skills/` 的 `references/`
   - 场景化执行经验、避坑步骤、验证清单：更新对应 `skills/<project>/<scene>/`
   - 数据结构、接口认知、外围事实补充：更新 `references/` 或索引清单

## 文档与代码同步原则

1. 代码发生有效变化后，相关文档原则上也应同步更新。
2. 若本次改动影响现有 skill、规范、接口说明、源码说明或架构说明，应将文档同步作为任务闭环的一部分。
3. 任何经验沉淀都必须以当前真实代码和本次任务事实为依据，禁止脱离代码现状主观扩写。

## 历史记录机制

1. 根目录 `change-history/` 用于记录各源码工程已完成任务的任务级变更历史。
2. 采用分项目结构：`change-history/<project>/YYYY-MM-DD-<task-type>-<slug>.md`。
3. 每次任务记录应参考 `change-history/TEMPLATE.md`。
4. 本目录默认不主动读取，仅在历史追溯和经验复用时按需读取。

## 适用范围

本文件适用于当前模板工作空间根目录及其全部子目录。
