## 2. 目标与背景

- 需求目标：参考 `obra/superpowers` 的流程型 skill 组织方式，检查并改善当前模板在方案确认、实施计划、执行验证和工作流可复用性方面的能力。
- 背景说明：当前仓库已经通过 `AGENTS.md` 规定方案确认、节点验收与资料沉淀机制，但 `skills/` 主要提供项目场景占位模板，尚未形成工作空间级的可执行流程资产与配套校验。
- 当前现状：模板目录结构校验可通过；当前校验脚本仅验证必需路径存在；仓库在本任务开始前为干净的 `main` 工作区。
- 已知约束：严格遵循“方案确认 -> 文档落地 -> 节点推进 -> 用户逐节点验收”；不得在节点未验收前连续推进后续节点。
- 不在本次范围：不引入特定业务线事实；不改造外部工具或插件安装机制；不复制参考仓库全部能力。

## 3. 方案内容

- 方案结论：保留当前工作空间偏重人工确认与资产沉淀的治理特色，借鉴 `superpowers` 将高频治理环节拆成工作空间级 workflow skills，并增强模板校验。
- 目标落点：
  - 建立工作空间级流程 skill 的设计与任务路由规则。
  - 后续落地五类核心 workflow skills：方案确认、实施计划、系统化排错、完成验证、分支工作流。
  - 将校验能力从目录存在性检查扩展到关键流程资产完整性检查，并清理模板卫生问题。
- 涉及模块 / 页面 / 服务：工作空间文档入口、skill 模板、校验脚本和模板基础文件。
- 涉及目录 / 文件：`docs/development-specs/`、`skills/`、`workspace-assets-index.md`、`scripts/`、`.gitignore` 及本方案文档。
- 预期影响范围：改善以本模板初始化的新工作空间内 AI 任务路由、流程执行一致性和模板验收可靠性。
- 风险点：流程资产过多可能增加阅读成本；因此 workflow skills 只覆盖通用门禁和高频流程，项目细节仍归属项目场景 skill。
- 方案调整规则：
  - 若方案、实施路径、节点拆分、验收方式或优先级发生调整，必须先与用户确认，再同步更新本文档。
