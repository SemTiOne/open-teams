# 编写实施计划

## 适用范围

用于方案已经获得用户明确确认、且任务将进入代码、配置、脚本或相关文档实施的阶段。

## 触发条件

- 用户明确回复同意方案、确认实施或验收通过后要求继续下一节点。
- 已确认方案需要拆解为可执行、可验证、可逐节点验收的步骤。

## 前置输入

- 已确认的方案结论与用户确认原话。
- `task-plans/TEMPLATE.md`。
- 目标仓库、目录、分支策略与当前工作区状态。
- 与实施节点直接有关的项目文档、场景 skill 和源码事实。

## 产物与退出条件

- 产物：`task-plans/` 下的方案与实施计划文档，包含节点状态、实施步骤、验证方法、验收标准和调整规则。
- 退出到实现阶段的条件：计划文档已落地，当前可执行节点已标为 `进行中`，且分支/工作区门禁已处理。
- 每个节点完成后必须回写结果并等待用户验收，不能自动跨节点继续。

## 工作流程

1. 提取确认过的目标、范围、风险和用户授权结论。
2. 按模板创建或更新任务计划文档，不读取无关历史计划。
3. 将实施拆成足够小且可验收的节点；为每节点写明文件范围、动作、验证和验收标准。
4. 对涉及源码或配置变更的任务，组合 `branch-and-worktree-workflow` 确认独立分支和已有改动风险。
5. 将当前待执行节点标为 `进行中`，再进入该节点实施。
6. 节点结束时组合 `verification-before-completion`，把完成说明、证据和遗留事项写回计划。

## 验证清单

- 计划是否记录了真实的用户确认结论。
- 节点是否足够具体，可以独立验证和验收。
- 当前节点状态是否与实际推进状态一致。
- 是否说明方案偏离时需要先确认并同步计划。

## 禁止事项

1. 不以对话中的口头计划替代 `task-plans/` 文档。
2. 不在计划文档落地前直接开始实施。
3. 不把多个尚未验收的节点合并推进。

## 按需资料

- `task-plans/TEMPLATE.md`：计划文档格式。
- `AGENTS.md`：方案记录与节点验收强约束。
- 对应项目场景 skill：节点实现和验证要求。
