# Architecture Review — 审查示例

## 示例 1：充分的方案（✅ 通过）

### 审查对象：引入 Redis 缓存层方案

**方案概述：** 在 API 服务和数据库之间增加 Redis 缓存层，缓存热点查询结果。针对读多写少场景。

**审查结论：** ✅ 通过

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 问题定义 | ✅ | 明确：P95 响应时间从 800ms 升至 2.5s，数据库 CPU >80% |
| 替代方案 | ✅ | 对比了：CDN 缓存 / 读写分离 / Redis 缓存 — 各有利弊分析 |
| NFR 覆盖 | ✅ | 扩展性（Redis Cluster）、可用性（Sentinel）、性能目标（P95 < 200ms） |
| 依赖评估 | ✅ | Redis 成熟度高、社区活跃、BSD 许可证、团队有运维经验 |
| 降级方案 | ✅ | Redis 故障时 fallback 到数据库直读 + circuit breaker |
| 成本 | ✅ | 预估新增 $200/月，vs 数据库扩容 $800/月 |

## 示例 2：不完整的方案（⚠️ 有条件通过）

### 审查对象：从 REST 迁移到 GraphQL

**方案概述：** 将所有 REST API 重写为 GraphQL，解决 over-fetching 问题。

**审查结论：** ⚠️ 有条件通过

| 问题 | 严重级别 | 说明 |
|------|---------|------|
| 缺少替代方案对比 | 🟠 High | 只提了 GraphQL 一个方案，未评估：加字段过滤参数、BFF 模式、或逐步迁移 |
| 未评估运维成本 | 🟠 High | GraphQL 的 N+1 查询、深度限制、复杂度分析——团队是否有经验？ |
| 无迁移计划 | 🟡 Medium | "一次性重写"风险极高。建议按模块逐步迁移，每个模块有回滚方案。 |
| 未说明降级 | 🟡 Medium | GraphQL 网关不可用时影响所有 API。建议保留 REST fallback 至迁移完成。 |

## 示例 3：有严重风险的方案（❌ 不通过）

### 审查对象：用户服务直接访问其他服务的数据库

**方案概述：** "为了快，用户服务直接读订单数据库获取用户订单数。"

**审查结论：** ❌ 需要重新设计

| 问题 | 严重级别 | 说明 |
|------|---------|------|
| 跨服务直接读数据库 | 🔴 Critical | 破坏了服务边界，订单数据库 schema 变更会影响用户服务。这是分布式单体反模式。 |
| 无降级方案 | 🔴 Critical | 订单数据库故障会级联影响用户服务。应有 API 调用 + circuit breaker。 |
| 数据一致性无保障 | 🟠 High | 绕过订单服务的业务逻辑，可能导致读到中间态数据。 |
| 安全边界模糊 | 🟠 High | 用户服务获得了订单数据库直接访问权限——权限爆炸。 |

**推荐替代方案：** 用户服务通过 API 调用订单服务 `GET /internal/orders/count?userId=X`，保持服务边界。如果性能是关键瓶颈，考虑在用户服务侧维护一个事件驱动的订单计数缓存。
