# Architecture Review Checklist

## 问题定义

- [ ] 要解决的问题是否清晰表述？（一句话能说清）
- [ ] 当前状态和约束是否明确？（现有系统、资源限制、时间限制）
- [ ] 不解决这个问题会有什么后果？（量化影响——成本、风险、速度）

## 方案设计

- [ ] 是否至少比较了 2 个替代方案（含"不做任何改动"）？
- [ ] 每个方案的核心取舍是否说清楚？（选择 A 得到的 vs 放弃的）
- [ ] 推荐方案是否有明确的理由（不是"感觉更现代"）？

## 非功能需求（NFR）

- [ ] 可扩展性：预期负载是多少？瓶颈在哪里？如何水平扩展？
- [ ] 可用性：单点故障在哪里？恢复时间（RTO）和恢复点（RPO）目标？
- [ ] 性能：关键路径的延迟目标是多少？P50/P95/P99？
- [ ] 安全：攻击面是什么？敏感数据如何保护？认证和授权模型？
- [ ] 成本：预计新增/节省多少基础设施成本？有没有更便宜的等价方案？
- [ ] 可维护性：团队是否有运维这个方案的能力？监控和日志是否覆盖？

## 集成与依赖

- [ ] 新增了什么外部依赖？（服务、库、数据库、消息队列）
- [ ] 每个新依赖的成熟度和社区活跃度？（避免孤立的、无人维护的依赖）
- [ ] 新依赖的许可证是否与项目兼容？
- [ ] 依赖失败时的降级策略是什么？（circuit breaker、fallback、graceful degradation）

## 数据

- [ ] 数据模型是否清晰？（实体、关系、约束）
- [ ] 数据一致性要求是什么？（强一致 vs 最终一致）
- [ ] 数据迁移计划是什么？（zero-downtime？回滚方案？）
- [ ] 敏感数据是否加密？（传输中 + 静态）

## 实施

- [ ] 是否可拆分为独立交付的里程碑？
- [ ] 第一个里程碑是什么？最低验证什么假设？
- [ ] 回滚计划是什么？如何判定需要回滚？
- [ ] 是否有明确的风险和缓解措施？

## 文档

- [ ] 决策是否记录为 ADR（Architecture Decision Record）？
- [ ] 系统图或时序图是否清晰？（一图胜千言）
- [ ] 是否有清晰的"不做什么"说明？（显式排除同样重要）
